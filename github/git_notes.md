# Git vs Github

Ένας συνοπτικός πίνακας για τις διαφορές ανάμεσα σε `Git` και `Github`.

| Χαρακτηριστικό | Git | GitHub |
|---|---|---|
| **Τι είναι** | Λογισμικό version control (τοπικό εργαλείο) | Διαδικτυακή πλατφόρμα φιλοξενίας αποθετηρίων |<!--  -->
| **Που τρέχει** | Στον υπολογιστή σου (offline) | Στο cloud (online) |
| **Σκοπός** | Παρακολούθηση αλλαγών κώδικα | Κοινή χρήση & συνεργασία πάνω σε κώδικα |
| **Εγκατάσταση** | Χρειάζεται εγκατάσταση (`apt install git`) | Δε χρειάζεται — πρόσβαση μέσω browser ή CLI |
| **Αποθήκευση** | Τοπικά στον δίσκο σου (`.git/` φάκελος) | Σε servers της Microsoft |
| **Χρήση χωρίς το άλλο** | Ναι — δουλεύεις αυτόνομα τοπικά | Όχι — το GitHub χρειάζεται Git |
| **Βασικές εντολές** | `commit`, `branch`, `merge`, `log` | `push`, `pull request`, `fork`, `clone` |
| **Κόστος** | Δωρεάν, open source | Δωρεάν με premium πλάνα |

**Με μια πρόταση:** Το **Git** είναι το εργαλείο, το **GitHub** είναι ο χώρος όπου η δουλειά σου ζει online και μοιράζεται με άλλους.

# Git Core

Το **git** χωρίζεται σε δύο βασικά μέρη:

- **Local** : Εδώ κάνουμε τις αλλαγές στον κώδικα τοπικά σε έναν υπολογιστή
    - **Working Directory**: Το τμήμα του **local**, στο οποίο κάνεις όλες τις αλλαγές
    - **Stage**: Όταν τελειώσεις ένα κομμάτι αλλαγών στο **working directory**, λες ότι είσαι έτοιμος να προχωρήσεις στο επόμενο κομμάτι
    - **Local Repository**: Εδώ κάνεις `commit`, δηλαδή αποθηκεύεις τις αλλαγές στο **Local Repository**
- **Remote**: Αφορά το κομμάτι του cloud το οποίο συνήθως αφορά το **github**.
    - **Push**: Η μεταφορά των αρχείων από το **Local Repository** στο **Remote**
    - **Pull**: Η μεταφορά από το **Remote** στο **Local Repository**.

### Ορ. Repository

- Με τον όρο **repository** εννοούμε ένα μέρος όπου όλα τα versions των αρχείων και το πλήρες ιστορικό αλλαγών είναι αποθηκευμένα.
    - Υπάρχουν 2 ειδών ένα **local** και ένα **remote**.

# Βασικές εντολές GIT CLI

- `git init`: Ενημερώνουμε το git ότι θα ξεκινήσει να δουλεύει σε αυτό τον φάκελο και φτιάχνει ένα νέο **working directory**.
    - Όταν τρέξεις αυτή την εντολή δημιουγείται το `.git` αρχείο. Αυτό το αρχείο αποθηκεύει όλες τις αλλαγές που γίνονται τοπικά.

- `git clone` + url: Αντιγράφει ένα **git repository** από το github στον υπολογιστή και δημιουργεί ένα νέο **workspace**.

- `git status`: Δείχνει το τι αλλαγές έχουν γίνει στο **workign directory**.
    -Οι αλλαγές με τις οποίες δεν έχει γίνει **commit** ακόμα.

- `git add`: Περνάμε από το **working directory** στο **stage**.
    - Με άλλα λόγια ενημερώνουμε το github ότι θέλουμε να κρατήσουμε τις αλλαγές.

    | Εντολή | Περιγραφή |
    |--------|-----------|
    | `git add --all` / `git add -A` | Stage όλα τα αρχεία του project (νέα, αλλαγμένα, διαγεγραμμένα) |
    | `git add .` | Stage μόνο τα αρχεία του τρέχοντος φακέλου και υποφακέλων |
    | `git add *` | Stage νέα ή αλλαγμένα αρχεία, **όχι** διαγεγραμμένα |
    | `git add <αρχείο>` | Stage μόνο το συγκεκριμένο αρχείο |

- `git reset`: Όλες οι αλλαγές που κάναμε με το add ακυρώνονται. Αλλά όχι τα διαγεγραμμένα αρχεία. 
    - `git reset --hard`: Επαναφέρει και τις αλλαγές που κάναμε σε αρχεία και τα διαγεγραμμένα αρχεία.

- `git commit -m <μήνυνα>`: Αφού ελέγξουμε ότι η αλλαγές που θέλουμε να κάνουμε είναι οι σωστές και υπάρχουν υα σωστά αρχεία στο **stage**, μπορούμε να κάνουμε **commit** που είναι η μόνιμη αποθήκευση των αλλαγών.

- `-git config --global user.email "<email>"`: Την πρώτη φορά χρήσης ορίζουμε το email μας

- `-git config --global user.name <όνομα>`: Ορίζουμε το όνομα χρήστη μας.
    - Και στα 2 μπορούμε να βάλουμε --local ώστε να ορίζονται μόνο για αυτό τον workspace.

- `git reset HEAD~`: Ακυρώνουμε το τελευταίο **commit** και όλα τα αρχεία περιμένουν να ξαναγίνουν staged.
    - `git revert <commit_id>`: Ακυρώνει τα τελευταία **commits**, μέχρι το *commit_id*.

- `git revert <commit_id>`: Δημιουργεί ένα νέο **commit**, το οποίο είναι κλώνος ενός παλιού **commit**. 
    - Ουσιαστικά μοιάζει με το **reset**, αλλά κρατάει το ιστορικό των αλλαγών.

- `git rm <αρχείο>`: Διαγράφει ένα αρχείο και το κάνει αμέσως **staged**. Αν το αρχείο περιέχει αλλαγές που δεν έχουν γίνει **commited** δεν μπορεί να διαγραφεί.
    - `git rm -f <αρχείο>`: Διαγράφει ένα αρχείο ακόμα και εάν είχε αλλαγές που δεν είχαν γίνει **commit**.
    - `git rm -cached <αρχείο>`: Αφαιρεί ένα αρχείο μόνο από το staged και όχι σαν φυσικό αρχείο.
    - `git rm -r <φάκελος>`: Διαγράφει τον φάκελο και ότι περιέχει αναδρομικά. Σε διαφορετική περίπτωση διαγράφει μόνο τον φάκελο.

- `git log`: Εμφανίζει το πλήρες ιστορικό των **commits**.
    - Για κάθε **commit** εμφανίζει ένα μεγάλο string. Αυτό είναι το **id** του commit με το οποίο μπορύμε να επιστρέψουμε σε εκείνο το version.
    - `git log --oneline`: Εμφανίζε το ιστορικό με τις βασικές πληροφορίες. Ακόμα και τα **ids** είναι πιο συμπυκνωμένα.
    - Για να βγείς από το **log view** πρέπει να πατήσεις `q`

### Branching

Κάθε project έχει ένα βασικό και επιμέρους **branches**. Το βασικό **branch** είναι το **main**, παλαιότερα **master**. 

Στο **master** γίνεται η ανάπτυξη του κώδικα, ωστόοσ όταν θέλουμε να δοκιμάσουμε ένα νέο feature, μπορούμε να φτιάξουμε ένα νέο branch, στο οποίο θα γίνει η δοκιμή, ώστε να μην πειράξουμε το main.

Με τον όρο **Merge** εννοούμε την  ένωση δύο **branches** (συνήθως μίας με την **main**),

- `git branch`: Εμφανίζει μία λίστα με όλα τα **branches**.
    - `git branch <όνομα>`: Δημιουργεί ένα νέο branch με ακριβώς την ίδια κατάσταση με το γονικό του που είναι το **active**, εκείνη την στιγμή.

- `git checkout <όνομα>`: Αλλάζει το **active branch**. 

- Ας υποθέσουμε ότι έχουμε δύο braneches, ένα **main** και ένα **development**.
    1. Αρχικά τα δύο αρχεία είναι ίδια
    2. Προσθέτουμε ένα αρχείο στο development (πχ test.txt)
    3. Αν κάνουμε `git checkout main` και πάμε στον φάκελο του workspace (από το σύστημα αρχείων) θα δούμε ότι το αρχείο δεν θα εμφανίζεται γιατί δεν είναι **active** το development branch στο οποίο ανήκει το αρχείο.

![Architecture](Diagramms/git_branches_workflow.drawio.svg)

Κάθε **commit** αφορά το **active branch**. 

- `git merge <όνομα> -m <Μήνυμα>`: Κάνουμε merge δύο branches. Η εντολή προφανώς εκτελείται από ένα **active branch**. Αυτό σημαίνει ότι στην ουσία μεταφέρουμε τις αλλαγές από το **branch** που ορίζουμε στο **active**. Δηλαδή έχει σημασία ποιο είναι το **active** και ποιο το **όνομα**. 
    - Υπάρχει και η τακτική να κάνουμε και το αντίστροφο, ώστε τα δύο branches να ταυτίζονται πλήρως
    - **SOS**: Πριν κάνουμε merge πρέπει να έχουμε κάνει commit όλες τις αλλαγές που θέλουμε.

### Merge Confilct

Με τον όρο **Merge Conflict** εννοούμε την περίπτωση όπου το ίδιο αρχείο άλλαξε και στα δύο **branches** και πάμε να κάνουμε **merge**.

Όταν συμβεί αυτό θα πρέπει να λύσουμε το **conflict** πριν καταφέρουμε το **merge**.

Όταν κάνουμε  **merge** και μας βγάλει ένα τέτοιο **error**, το git θα προσθέσει αυτόματα στο αρχείο που είναι το **conflict**, από το **active branch**, κάποια στοιχεία που θα δείχνουν που ακριβώς είναι το πρόβλημα. Εκεί μπορούμε να επιλέξουμε ποιο θέλουμε διαγράφοντας το άλλο. Έτσι το **git** καταλαβαίνει ότι πήραμε αυτή την απόφαση. Eπίσης θα διαγράψουμε και τον κώδικα του **git**.

### Αλλαγή commits

- `git checkout <commit_id>`: Επιστρέφουμε σε μία παλαιότερη εκδοχή του κώδικα μέσα από το id. 
    - Εάν υπάρχουν **untracked** ή **uncommited** αρχεία δεν θα επιτρέπει να κάνουμε **checkout**.
    - `git checkout <όνομα branch>`: Με αυτό τον τρόπο επιστρέφουμε στο τελευταίο **commit** του **branch**. 

- `git diff <commit_id1> <commit_id2>`: Εμφανίζει τις αλλαγές που υπάρχουν ανάμεσα σε δύο **commits**.
    - *Green*: Είναι αυτά που προστέθηκαν
    - *Red*: Είναι αυτά που αφαιρέθηκαν
    - **SOS**: Έχει σημασία η σειρά με την οποία βάζουμε τα δύο **commits**. Τυπικά βάζουμε το πιο νέο πρώτα ώστε να δείχνει τις αλλαγές από τη δική του "οπτική".

### Επαναφορά αρχείων - `restore`

Ας υποθέσουμε ότι κάνουμε μία αλλαγή αρκετά μεγάλη την οποία θέλουμε να αναιρέσουμε και δεν έχουμε κάνει ακόμα **commit**. Τότε αρκεί να κάνουμε **restore**.

- `git restore <αρχείο>`: Επαναφέρει ένα αρχείο στην κατάσταση που ήταν όταν κάναμε το τελευταίο **commit**. 
    - Μπορούμε να βάλλουμε και φακέλους και θα κάνει **restore** όλο τον φάκελο.
    - `git restore .`: Κάνει **restore** όλο το **repository**.
    - `git restore --staged <αρχείο ή .>`: Αφαιρεί τα αρχεία από το **stage** αλλά τα διατηρεί στο **working directory**.



### Προσωρινή αποθήκευση - `stash`

Υπάρχει η περίπτωση να είσαι στο ενδιάμεσο ενός μεγάλου project και να χρειάζεται να αλλάξεις **branch**. Ωστόσο ακόμα δεν είναι έτοιμο για **commit**.

- `git stash`: Αποθηκεύεις προσωρινά όλες τις αλλαγές σε ένα **branch**, και έπειτα μπορείς να αλλάξεις άφοβα **branch**
    - **SOS**: Εάν έχεις κάνει αλλαγές που δεν έχουν γίνει **commit** ή **stash** και πας να αλλάξεις **branch**, θα εμφανίζει σφάλμα.
    - Μόλις κάνουμε **stash** οι αλλαγές εξαφανίζονται από **working directory**, ωστόσο παραμένουν αποθηκευμένες.

- `git stash pop`: Αφαιρεί την πιο πρόσφατη **stashed** δουλειά από το **stash** και την επαναφέρει.

- `git stash apply`: Είναι σαν το `pop`, αλλά κρατάει την αλλαγή στο `stash`.

- `git stash list`: Εμφανίζει όλο το `stash`.
    - Κάθε στοιχείο έχει και ένα όνομα (πχ `stash@{0}`) το οποίο μπορούμε να χρησιμοποιήσουμε για να κάνουμε **pop** ή **apply**, συγκεκριμένες εντολές.

- `git stash drop`: Απλά διαγράφει το τελευταίο στοιχείο από το `stash`.

### Αλλαγή Βάσης - Rebase

Αλλαγή βάσης κάνουμε από ένα **branch** σε ένα άλλο. Ας υποθέσουμε `main` και `feature`, σε αυτή την περίπτωση εάν κάνουμε **rebase**, από το `feature` στο `main`, κάθε **commit** που κάναμε στο `main` γίνεται και στο `feature`. 

- `git rebase main`: Θεωρώντας ότι είμαστε στο **branch** `feature`.

Η λογική από πίσω είναι η εξής:
1. Το **git** βρίσκει το τελευταίο κοινό **commit**.

2. Όλα τα **commits** στο **active branch**, (εδώ το `feature` αποκόπτονται προσωρινά

3. Το **git** κάνει **commit** τα **commits** του `main`

4. Επαναφέρει τα **commits** του `features`.

Γενικά η λογική είναι παρόμοια με το **merge**. Από την μία το **rebase** είναι προτιμότερο γιατί κάνει το **commit log** πιο καθαρό από ότι το **merge**. Από την άλλη όμως θέλει προσοχή εάν κάποιος κάνει δουλειά στο ίδιο branch. 
- **SOS:**Όταν κάνουμε **rebase** αλλάζουν τα **commit ids** στο **branch**, με αποτέλεσμα ο άλλος να έχει πρόβλημα. Όταν λοιπόν κάνουμε **rebase** πρέπει να ενημερώνουμε και να είναι ξεκάθαρο.

### Squashing - Ένωση 2 ή περισσότερων commits

Για να πετύχουμε την ένωση δύο ή περισσότερων **commits** υπάρχουν 2 βασικοί τρόποι:

- **Interactive Rebase** -> Για ενώσεις ενδιάμεσων **commits**
    1. `git rebase -i HEAD~n`, όπου n ο αριθμός των commits να δει πίσω το **git**.
    2. Στον **editor** θα ανοίξει αυτόματα ένα αρχείο που θα έχει 2 γραμμές που ξεκινάνε με `pick`. Το δεύτερο `pick` θα το κάνεις `squash` ή απλά `s`.
    3. Αποθήκευσε και κλείσε το αρχείο
    4. Θα ανοίξει ένα νέο αρχείο στο οποίο θα προσθέσεις ένα νέο μήνυμα για το κοινό **commit**. Το αποθηκεύεις και κλείνεις.

- **Reset** -> Μόνο για τα τελευταία **commits**
    1. `git reset --soft HEAD~n` -> Ακυρώνει τα τελευταία n commits αλλά κρατάει τις αλλαγές που έχουν γίνει τελευταία.
    2. `git commit -m <μήνυμα>` -> Κάνουμε απλό commit. 

# Remote Repository - Github

- `git push origin main`: Στέλνει όλες τις αλλαγές που έχουν γίνει **commit** στο "main" branch, στο github.
    - Δεν θα στείλει τα άλλα branches.

- `git fetch`: Φαίρνει όλες τις αλλαγές από το github στο local repository. Ωστόσο για να είναι εμφανείς αυτές οι προσθήκες πρέπει να κάνουμε **merge**.

- `git merge`: Κάνει **merge** τις αλλαγές από το github στο local repository.

- `git pull`: Κάνει **fetch** και **merge** ταυτόχρονα στο local repository.

![Architecture](Diagramms/pull_eq.drawio.svg)

### Pull Request

Με τον όρο **Pull Request** εννοούμε την αίτηση για να γίνει **merge** ένα **branch** μας συνήθως στο **main branch**. 

Με άλλα λόγια ζητάμε να κάνουμε αλλαγές στο **repository** κάποιου άλλου.


# VS CODE + Git

Στην μπάρα κάτω δεξιά εμφανίζεται το όνομα του **branch** στο οποίο εργαζόμαστε.

**SOS**: Οι εντολές που αναφέρονται είναι μέσα της **command pallete**, στην οποία έχουμε πρόσβαση πατώντας `ctrl + shift + p`.

- `git rename branch`: Μας επιτρέπει να αλλάξουμε το όνομα του **branch** στο οποίο βρισκόμαστε.

- `git Merge`: Κάνει **merge**.

- `git Create Branch`: Δημιουργεί ένα νέο **branch**.

- Στο **Source Control** tab εμφανίζονται οι αλλαγές που έχουμε κάνει στα αρχεία του **working directory**.
    - Με το σύμβολο `U` είναι τα αρχεία που είναι **untracked**, τα οποία μπορούμε να κάνουμε **staged**, πατώντας το `+`.
    - Με το σύμβολο `Α` είναι τα αρχεία που είναι **staged**, τα οποία για να κάνουμε **commit**, αρκεί να γράψουμε το μήνυμα και να πατήσουμε το **commit**.
    - Με το σύμβολο `M` είναι τα αρχεία που έχουν γίνει **modified**.  

- Στο πλάι του κώδικα αμέσως μετά τα νούμερα υπάρχουν κάποιες μπάρες που δηλώνουν τις αλλαγές που έχουμε κάνει στον κώδικα.

    | Χρώμα | Σημασία |
    |-------|---------|
    | Πράσινη | Προσθέσαμε γραμμή κώδικα |
    | Κόκκινη | Αφαιρέσαμε γραμμή κώδικα |
    | Μπλε | Αλλάξαμε γραμμή κώδικα |

- Πατώντας σε ένα αρχείο από το  **Source Control** γίνεται αυτόματα το diff σε ένα αρχείο.
